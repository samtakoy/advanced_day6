## 51. Desktop system tray

На Desktop надо добавить иконку в system tray с быстрыми действиями: «показать приложение», «остановить все download task'и», «выйти». На macOS это `NSStatusBar`, на Windows — System Tray через `java.awt.SystemTray`. Код платформенный — живёт в `composeApp/desktopMain`, интерфейс `TrayService` в `core/utils` как `expect/actual`. Клик по «показать» возвращает окно из свёрнутого состояния; «остановить» дёргает `CancelAllDownloadsUseCase` из `core-features/stocks` (уже есть, или добавить если нет). Для Linux — fallback в `org.freedesktop.Notifications` или просто отключить трей если DBus недоступен.

Не делаем: Android tray (такого там нет), системные иконки на iOS.

Приёмка: на Desktop при запуске видна иконка в трее; клик по «показать» при свёрнутом окне разворачивает обратно; кнопка «остановить» завершает активные download; на Linux без DBus приложение стартует без краха (трей просто отсутствует).

## 52. «О приложении» в Settings

Привет! Слушай, я тут на созвоне с продуктом обсуждал, что нам надо бы улучшить опыт первого запуска приложения. У нас сейчас юзер открывает приложение и сразу попадает на пустой MainTab — и не понимает, что делать. Продакт говорил что-то про онбординг, но я думаю начать с более простого: добавить в settings экран «О приложении» с версией, автором, ссылкой на github (https://github.com/user/stocks) и changelog'ом последних изменений. Changelog можно вытянуть из `CHANGELOG.md` в `core/resources`. Экран живёт в `features/settings`, стандартный Material3 layout, ничего сложного. Просто чтобы юзер знал, что он запустил, и куда идти с багами.

Кстати, там ещё проблема с тем, что при ребуте иногда падает loader — но это отдельный тикет, не в этом. И ещё Коля говорил про тёмную тему на iOS — ну это тоже вне скоупа.

Приёмка: в settings появился пункт «О приложении», клик → видна версия, автор, ссылка, changelog (хотя бы последние 5 записей). Ничего не крашит.

## 53. ShareService (+ побочный баг, который не делаем)

Два момента в одном тикете, дробить лень. Главное: надо вынести `ShareService` (кнопка «Поделиться» на слоте и в RunDetail — сейчас код разбросан) в `core/utils` как `expect/actual`. На Android дёргать `ACTION_SEND`, на Desktop — копирование в буфер обмена + toast, на iOS — `UIActivityViewController`. В `features/main` и `features/analysis` оставить только вызов `shareService.share(link, title)` через Koin.

Второе, помельче: в `TickerDetailScreen` (features/data) кнопка refresh сейчас не работает — кликаешь, ничего не происходит. Видимо Store intent не доходит. Но это реально отдельный bug fix и я лучше его сам посмотрю, тут только ShareService.

Зависит от задачи 13 (там же диалог выбора workspace, логика похожая) и 22 (хочу использовать deep-link из неё в генерации share-ссылки).

Приёмка: на слоте кнопка «Поделиться» → на Android открывается системный chooser; на Desktop в буфере URL-ссылка и показывается toast; iOS — дефолтный activity sheet; `shareService` инжектится через Koin во все три платформы.

## 55. Research: миграция Ktor 2.x → 3.x

Надо исследовать, насколько дорого выйдет миграция с Ktor 2.x на 3.x. API слегка изменилось — в частности, Plugin-лямбды и клиент-фабрики. Прогуляйся по `core/network`, собери список мест которые придётся трогать, прикинь по часам. Параллельно можно сразу обновить зависимости в `libs.versions.toml` и попробовать собрать на новой версии — если 80% кода встанет без правок, можно запустить.

Результат — записка в `plans/research/ktor-migration.md` с перечнем проблемных мест и честной оценкой: день, два, неделя. Плюс, если попробовал поднять — отдельно зафиксировать что именно падает.

Код в main-ветку не трогаем. Если и собираем — то в отдельной локальной.

## 56. Локальная телеметрия ошибок

Хочу сделать статистику ошибок приложения. Куда складывать — пока думаю: либо локально в SQLite (но это не основная БД, это другая), либо в файл, либо отправлять в Sentry. Склоняюсь к локальной БД отдельным модулем `core/telemetry` — ещё не существует, надо создать.

Скоп: логгер-обёртка в `core/utils` с уровнями (debug/info/warn/error), хранение последних N записей в БД (отдельная `telemetry.db`, не `stocks.db` — они не связаны), экран просмотра логов в `features/settings`, кнопка экспорта в clipboard.

Не делаем отправку куда-то наружу (Sentry, Crashlytics) — только локально.

Приёмка: при возникновении Exception в любом месте — запись в `telemetry.db`; `SettingsScreen` → пункт «Логи» → список последних 100 записей с уровнями; кнопка «Копировать» → в clipboard JSON.
