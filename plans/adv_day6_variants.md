# Варианты архитектуры для fine-tune агента-стейт-машины

> Все три варианта реализуют одну и ту же идею пользователя (план во внешнем хранилище, task_id-скоупинг, пошаговое исполнение с self-check), но различаются по инфраструктуре, сложности и тому, как именно это отражается в датасете для fine-tune.

---

## Вариант A. MCP Memory Layer (канонический)

### Идея в двух строках
Отдельный **MCP-сервер** хранит state каждой задачи (план, шаги, статусы) в своей памяти, скопированно по `task_id`. Fine-tune-нутая LLM общается с этим сервером через именованные семантические инструменты — без прямого контакта с файловой системой.

### Архитектура (вербально)

```
┌────────────┐   messages   ┌────────────┐   MCP tool calls   ┌──────────────────┐
│   User     │ ───────────► │   LLM      │ ─────────────────► │  MCP Memory      │
│  (runner)  │ ◄─────────── │(fine-tuned)│ ◄───────────────── │  Server          │
└────────────┘              └────────────┘    tool results    │  - task registry │
                                                              │  - step store    │
                                                              │  - SQLite/in-mem │
                                                              └──────────────────┘
```

Основные MCP-инструменты (семантические, высокоуровневые):
- `task_create(goal) → task_id` — регистрирует задачу
- `plan_write(task_id, steps[])` — сохраняет весь план сразу
- `step_read(task_id, step_n) → {goal, preconds, checklist, action_hint}` — читает текущий шаг
- `step_update_result(task_id, step_n, {status, notes, artifacts})` — фиксирует исход
- `task_status(task_id) → {current_step, total, done_steps[]}` — дашборд для LLM
- `plan_revise(task_id, from_step, new_tail[])` — обновляет хвост плана при replan

### Стек / библиотеки
- **MCP SDK**: официальный `@modelcontextprotocol/sdk` (TypeScript) или `mcp` (Python). На выбор.
- **Storage**: SQLite (`better-sqlite3` или `sqlite3`) — `tasks(id, goal, status)`, `steps(task_id, n, content, result, status)`.
- **LLM client**: OpenAI Python SDK, tools экспонируются через function calling (MCP можно адаптировать в OpenAI tools).
- **Язык сервера**: Python (быстрее для учебной задачи) или Kotlin + `kotlinx-mcp` если хочется в стеке проекта.
- **Для Дня 6**: достаточно мока `InMemoryStore` без SQLite — MCP-сервер не обязателен, нужны лишь сигнатуры для датасета.

### Принципы
1. **Semantic tools > filesystem tools** — LLM не знает про файлы, она оперирует концепциями "шаг", "задача", "статус". Это уменьшает поверхность галлюцинаций (меньше произвольных параметров).
2. **Инкапсуляция формата** — MCP-сервер сам решает, как физически хранить шаг (markdown, JSON, запись в БД). LLM это не касается.
3. **Изоляция через task_id** — параллельные задачи не мешают друг другу, легко логировать и воспроизводить.
4. **Read-before-write дисциплина** — fine-tune прошивает обязательный `step_read` перед `step_update_result`.

### Как выглядит строка JSONL для этого варианта
```
system: "Ты агент-исполнитель. Используй MCP tools для работы со state. Всегда читай шаг перед выполнением."
user: "Задача: добавь логгер в класс AuthService. task_id=t-0042."
assistant (tool_call): {"name":"step_read","args":{"task_id":"t-0042","step_n":1}}
tool: {"goal":"Найти файл AuthService","preconds":[],"checklist":["путь подтверждён list_dir"],"action_hint":"list_dir на shared/"}
assistant (tool_call): {"name":"step_update_result","args":{"task_id":"t-0042","step_n":1,"status":"in_progress"}}
assistant (text): "THOUGHT: ... ACTION: list_dir(...)"
...
```

### Плюсы
- **Продакшен-ready**: тот же MCP можно подключить к Claude Desktop, Cursor, любому MCP-клиенту.
- **Переиспользуемость**: один сервер — много агентов.
- **Чистая абстракция**: LLM не тонет в деталях файловой системы.
- **Хорошо измеряется**: логи MCP = готовый трейс для отладки.

### Минусы
- **Инфраструктурный overhead**: нужно поднять MCP-сервер, даже если моковый.
- **Датасет сложнее**: multi-turn с tool_calls и tool_results, строже формат OpenAI `messages`.
- **На Дне 6 придётся иметь хотя бы стаб** — иначе непонятно, какие tool signatures class модели.

### Когда выбрать
Если есть намерение развивать агента дальше (Дни 7-10+), переиспользовать в Claude Desktop, или если нравится чистая архитектура с намёком на продакшен.

## Вариант B. Файловый scratchpad + OpenAI Function Calling

### Идея в двух строках
Никакого MCP — модель напрямую дёргает низкоуровневые FS-tools (`write_file`, `read_file`, `list_dir`). State — это обычные файлы в папке `.tasks/<task_id>/`. Проще, прозрачнее, всё видно глазами.

### Архитектура (вербально)

```
┌────────────┐   messages   ┌────────────┐   function calls   ┌──────────────────┐
│   User     │ ───────────► │   LLM      │ ─────────────────► │  Local Python    │
│  (runner)  │ ◄─────────── │(fine-tuned)│ ◄───────────────── │  executor        │
└────────────┘              └────────────┘     results        │                  │
                                                              │  ↓ доступ        │
                                                              │  .tasks/         │
                                                              │    t-0042/       │
                                                              │      plan.md     │
                                                              │      01_step.md  │
                                                              │      02_step.md  │
                                                              └──────────────────┘
```

Инструменты — стандартные FS (через function calling):
- `write_file(path, content)` 
- `read_file(path)`
- `list_dir(path)`
- `append_file(path, content)` — для апдейта статусов

Модель учится шаблонам путей: `.tasks/{task_id}/plan.md`, `.tasks/{task_id}/{n:02d}_step.md`, `.tasks/{task_id}/status.json`.

### Стек / библиотеки
- **LLM**: OpenAI SDK, function calling (встроенный механизм, MCP не нужен).
- **Executor**: тонкий Python-скрипт (~100 строк), принимает tool calls, делает FS-операции в sandbox-папке.
- **Sandbox**: `tempfile.TemporaryDirectory()` или namespaced `.tasks/<uuid>/`.
- **Нечего ещё ставить** — только `openai` пакет.

### Принципы
1. **Transparency first** — всё видно в файловой системе, дебаг через `ls` и `cat`.
2. **Соглашения важнее API** — дисциплину даёт fine-tune + строгая структура путей, а не типизированные инструменты.
3. **Task scope = папка** — task_id материализуется как папка, её удаление = cleanup.
4. **Minimal dependency surface** — меньше зависимостей, меньше ломающихся вещей.

### Как выглядит строка JSONL для этого варианта
```
system: "Ты агент. Для task_id ВСЕГДА работаешь в папке .tasks/<task_id>/. Перед действием читай текущий шаг."
user: "Задача: добавь логгер в AuthService. task_id=t-0042. Сейчас читай плановый шаг 1."
assistant (tool_call): {"name":"read_file","args":{"path":".tasks/t-0042/01_step.md"}}
tool: "# Step 1\nGOAL: найти AuthService\nCHECKLIST: - путь подтверждён\nACTION_HINT: list_dir shared/"
assistant (text): "THOUGHT: ... ACTION: list_dir(shared)"
assistant (tool_call): {"name":"list_dir","args":{"path":"shared/"}}
...
```

### Плюсы
- **Минимальная инфраструктура** — ноль серверов.
- **Лёгкий дебаг** — открыл папку задачи, видишь всё состояние.
- **Быстрый старт** — за полдня собирается рабочий стенд.
- **Датасет проще** — tool calls меньше, меньше multi-turn магии.

### Минусы
- **Галлюцинация путей** — модель может "случайно" прочитать не тот файл или вылезти за scope. Надо либо обёрткой запрещать выход за `.tasks/<task_id>/`, либо жёстко учить на датасете.
- **Меньше семантики** — LLM оперирует путями и содержимым файлов, "задача" и "шаг" — это соглашения, а не типы.
- **Продакшен-путь чуть длиннее** — чтобы подключить к Claude Desktop, всё равно придётся завернуть в MCP.
- **Риск загрязнения FS** — если не делать sandbox-изоляцию, папки будут копиться.

### Когда выбрать
Если хочется **минимум инфраструктуры**, цель — именно эксперимент Дня 6 с fine-tune, и интеграция в Claude Desktop / Cursor не в приоритете. Идеально, чтобы пощупать концепцию за день.

## Вариант C. Conversation-only state (без внешнего хранилища)

### Идея в двух строках
Никакой памяти за пределами самого чата. Модель в каждом своём ответе выдаёт **полный структурированный state-блок** (текущий план, номер шага, статусы). "Память" = история сообщений + самодисциплина модели ретранслировать state.

### Архитектура (вербально)

```
┌────────────┐   messages (contain state)   ┌────────────┐
│   User     │ ────────────────────────────► │   LLM      │
│  (runner)  │ ◄──────────────────────────── │(fine-tuned)│
└────────────┘    assistant message          └────────────┘
                  contains STATE block +
                  next ACTION
```

Формат ответа модели (пример):
```
<STATE task_id="t-0042">
  plan:
    1. [DONE] список файлов shared/
    2. [NOW]  прочитать AuthService.kt
    3. [TODO] вставить логгер
    4. [TODO] verify
  current: 2
  checklist_for_current:
    [ ] файл существует
    [ ] имя класса совпадает
</STATE>
<ACTION>{"tool":"read_file","args":{"path":"shared/AuthService.kt"}}</ACTION>
```

Runner просто передаёт tool_result обратно, и модель в следующем ответе снова издаёт свежий `<STATE>` с обновлёнными статусами + новый `<ACTION>`.

### Стек / библиотеки
- **LLM**: OpenAI SDK (function calling опционально — можно даже парсить `<ACTION>` блоки самому).
- **Хранилище**: нет. Всё в `messages[]`.
- **Парсер**: простой regex / XML-парсер для `<STATE>` и `<ACTION>`.
- **Tools**: любые — LLM просто просит runner-а выполнить, runner вставляет результат как user message.

### Принципы
1. **State is a first-class part of the output** — fine-tune учит модель ВСЕГДА начинать с `<STATE>` и заканчивать `<ACTION>`.
2. **Self-referencing** — модель читает своё же предыдущее состояние в истории и обновляет его.
3. **No external contract** — модель может работать где угодно, не завязана на MCP/FS.
4. **Простейшая интеграция с fine-tune** — OpenAI JSONL практически без multi-turn tool magic, одна модель, одна structured output-схема.

### Как выглядит строка JSONL для этого варианта
```
system: "Каждый ответ начинается с <STATE task_id=...> блока и заканчивается <ACTION> JSON."
user: "Задача: добавь логгер в AuthService. task_id=t-0042."
assistant: "<STATE task_id=\"t-0042\">\nplan:\n  1. [NOW] list_dir shared/\n  ...\n</STATE>\n<ACTION>{\"tool\":\"list_dir\",\"args\":{\"path\":\"shared/\"}}</ACTION>"
user: "tool_result: [AuthService.kt, UserService.kt]"
assistant: "<STATE task_id=\"t-0042\">\nplan:\n  1. [DONE] list_dir shared/\n  2. [NOW] read_file AuthService.kt\n  ...\n</STATE>\n<ACTION>..."
```

### Плюсы
- **Нулевая инфраструктура** — только LLM и парсер ответов.
- **Самый быстрый путь к работающему прототипу** — часы, не дни.
- **Fine-tune формат прост** — structured text в `assistant`, никаких tool_calls вперемешку.
- **Отличная тренировка structured output** — модель учится стабильно выдавать формат.

### Минусы
- **Контекст растёт линейно** — на длинных задачах "забудет хвост". Это именно та проблема, которую мы изначально хотели решить! Частично решается через `<STATE>` (свежее состояние в каждом ответе), но история всё равно в окне.
- **Нет persistence между сессиями** — если процесс упал, состояние теряется (можно хранить JSON снэпшоты, но это уже полу-шаг к варианту B).
- **Нет параллельности** — task_id используется внутри, но реально это просто тег, изоляции между задачами нет.
- **LLM тащит груз ретрансляции** — на слабой модели это НЕ ноль-стоимость: ошибка в копировании state = баг.

### Когда выбрать
Если цель — **минимально демонстрируемая идея за самый короткий срок**, или если дата-сет нужен для обучения именно паттерну structured output, а не tool использования. Хороший старт перед переходом к B или A.

## Сравнительная таблица

| Критерий | A: MCP Memory Layer | B: Файловый scratchpad | C: Conversation-state |
|---|---|---|---|
| **Сложность реализации** | Высокая (нужен сервер) | Средняя (100 строк Python) | Низкая (парсер) |
| **Инфраструктура** | MCP SDK + SQLite | Только OpenAI SDK | Только OpenAI SDK |
| **Persistence** | Есть (БД) | Есть (файлы) | Нет (в окне контекста) |
| **Изоляция по task_id** | Отличная | Хорошая (папки) | Символическая |
| **Масштаб длинной задачи** | Отлично — контекст мал | Отлично — контекст мал | Плохо — контекст растёт |
| **Сложность датасета** | Высокая (multi-turn с tool_calls + results) | Средняя (tool_calls, но меньше) | Низкая (structured text) |
| **Дебаг** | Через логи MCP | Через `ls`/`cat` папок | Через чтение истории |
| **Переиспользование** | Плюс — любой MCP client | Только свой runner | Только свой runner |
| **Подходит для слабой LLM** | Да (минимум per-step контекста) | Да (минимум per-step контекста) | Частично (контекст растёт) |
| **Время до первого запуска** | 2-3 дня | 1 день | Полдня |
| **Риск галлюцинаций путей** | Низкий (семантические tools) | Средний (нужна дисциплина) | Низкий (state в ответе, не в tools) |
| **Production-path** | Прямой | Через обёртку в MCP | Нужен серьёзный рефакторинг |

## Рекомендация

**Для задачи Дня 6 оптимален Вариант B (файловый scratchpad)** со следующими соображениями:

1. **Реалистично за день**: скелет runner-а + 50 JSONL примеров + валидатор собираются без блокеров.
2. **Хорошо показывает идею**: task_id = папка, шаги = файлы — всё визуально понятно, можно показать на видео.
3. **Решает главную проблему** слабой LLM: на каждом шаге в контексте только один маленький файл шага.
4. **Эволюционирует в A**: когда появится потребность в MCP-интеграции, файловый runner оборачивается в MCP-сервер за пару часов — tool names те же.
5. **Подходит для Дней 7-10**: вариант C потребует переделки, а B — нет.

**План действий на День 6:**
1. Зафиксировать tool signatures (4-5 инструментов: `write_file`, `read_file`, `list_dir`, `append_file`, опционально `delete_file`).
2. Спроектировать формат файла шага (markdown или JSON с полями `goal`, `preconds`, `checklist`, `action_hint`, `success_criteria`).
3. Написать meta-prompt для генерации 40 синтетических примеров (на GPT-4o / Claude 3.5 Sonnet) — с разбивкой по сценариям: нормальный путь, ошибка + replan, недостаток данных + QUESTION, multi-file refactor.
4. Добавить 10 реальных примеров (можно из собственных задач по KMP — то, что уже обсуждено в `adv_day6_plan.md`).
5. Написать валидатор: валидный JSON, 3 роли, непустые content, наличие task_id во всех tool calls, обязательный `read_file` шага перед экшеном.
6. Снять baseline на 10 eval-примерах через `gpt-4o-mini` без fine-tune. Зафиксировать метрики (см. раздел 7 `adv_day6_analisys.md`).
7. Написать upload-client: `openai files.create` + `openai fine_tuning.jobs.create`, с polling. Не запускать.

**Вариант A — следующий шаг**: после успеха fine-tune на B, обернуть runner в MCP-сервер для интеграции с Claude Desktop и реальным кодинг-ассистентом.

**Вариант C — не брать**: проблема растущего контекста противоречит самой цели задачи (помочь слабой модели удерживать длинные планы).
