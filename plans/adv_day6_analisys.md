# Анализ задачи "День 6: fine-tune слабой LLM под агента-стейт-машину"

## 1. Цель пользователя (своими словами)

Обучить **слабую модель** (типа `gpt-4o-mini` или её дообученная версия) вести себя как **дисциплинированный пошаговый агент**:

1. Получив задачу — **сначала думать**, а не сразу бросаться делать.
2. Составить **детальный план** и **разбить его на атомарные шаги**.
3. **Вынести план во внешнее хранилище** (MCP memory layer), привязанное к **уникальному task_id**.
4. На этапе исполнения **не полагаться на свой контекст**, а перечитывать файл текущего шага перед каждым действием.
5. В каждый шаг **заранее вшить чек-лист самопроверки** (что учесть, что проверить после).
6. Пройти все шаги как **стейт-машина**: read step → act → self-check → mark done → next step.

Ключевая идея: **не усиливаем «мозги» модели — усиливаем её инфраструктуру**. Всё, что слабая LLM не может удержать в голове, выносится наружу и подаётся ей порциями в нужный момент.

---

## 2. Почему это реально улучшит слабую LLM (главный вопрос)

Слабая модель проваливается на трёх вещах:

| Слабость слабой LLM | Что ломается без архитектуры | Что чинит наша архитектура |
|---|---|---|
| **Короткий эффективный контекст** (даже если окно 128k, attention "размывается" после 4-8k) | Модель "забывает" первый шаг плана к десятому | На каждом шаге читается только файл текущего шага (~200-500 токенов) |
| **Плохая рабочая память / "думание в голове"** | Пропускает self-check, не связывает действие с общим планом | Чек-лист материализован в файле — модель не может "забыть" его |
| **Склонность к improvization** (ответила что попало, заполнив "пробелы") | Галлюцинирует пути, имена, версии, параметры | Fine-tune прошивает: "сначала list_dir/read_step, потом action" — не даёт пропустить этап |

### Почему именно fine-tuning, а не prompt-инжиниринг?

Prompt задаёт **инструкцию**. Fine-tune задаёт **распределение вероятностей выходов**. Для слабой модели это критично:

- В prompt можно написать "всегда начинай с чтения файла шага" — но слабая модель в 30% случаев этого не сделает, потому что в её обучающей выборке такого паттерна почти не было.
- После fine-tune на 50 примерах, где **каждый** ответ начинается с `read_step` или с блока `THOUGHT`, модель математически "проваливается" в этот паттерн — softmax распределение смещено туда.
- Это и есть "локальный аттрактор" из лекции: создаём в пространстве весов яму, в которую модель скатывается естественно, без уговоров.

### Побочный эффект: экономия токенов

После fine-tune системный промпт можно **сократить в 5-10 раз** — модель уже "знает" правила и не нуждается в их ежекадровом напоминании. Для `gpt-4o-mini` в продакшене это прямая экономия денег и latency.

## 3. Мозговой штурм подходов (базовые идеи, без деталей)

Чтобы дать пользователю пространство выбора, вот спектр возможных решений — от простого к сложному:

### 3.1. In-context state (без внешнего хранилища)

Модель в каждом ответе переиздаёт **полный state-блок** в структурированном виде (например, JSON). План, текущий шаг, статус — всё живёт в сообщениях истории чата.

- **+**: нулевая инфраструктура, только OpenAI API.
- **−**: контекст растёт линейно от числа шагов → для слабой модели это именно та проблема, которую мы хотели решить. На 10+ шагах модель начнёт терять фокус.

### 3.2. Файловый scratchpad (FS-tools, без MCP)

Модель дёргает `write_file`, `read_file`, `list_dir` через **обычный function calling**. Пишет шаги в `.tasks/<task_id>/01_step.md` и т.д.

- **+**: просто, прозрачно, всё можно дебажить глазами.
- **−**: реальные файлы на диске → нужна уборка, риск затереть чужое, нет изоляции между запусками.

### 3.3. MCP Memory Layer (canonical)

Отдельный **MCP-сервер** с in-memory (или SQLite) хранилищем, экспонирует семантические инструменты: `plan_create(task_id)`, `step_read(task_id, n)`, `step_update(...)`, `step_mark_done(...)`, `task_status(task_id)`.

- **+**: чистая изоляция по task_id, переиспользуется между разными агентами, масштабируется до продакшена.
- **−**: нужно написать сам MCP-сервер (хотя есть готовые SDK — официальный Python/TS SDK от Anthropic).

### 3.4. Готовые memory-библиотеки (Mem0, LangMem, Zep)

Фреймворки для LLM-памяти с векторным retrieval и фактологическим хранилищем.

- **+**: всё уже сделано, есть готовые интеграции.
- **−**: оверкилл для задачи шагового плана (они про долгосрочную память ассистента, а не про текущий task state), закладывает зависимость от сторонней платформы. Для учебной задачи Дня 6 — не по размеру.

### 3.5. OpenAI Assistants API / Threads

Thread ID = task ID. История ведётся платформой.

- **+**: встроенная персистентность, минимум кода.
- **−**: vendor lock-in в OpenAI Assistants, сам thread это всё тот же растущий контекст (проблема 3.1).

### 3.6. Agent frameworks (LangGraph, CrewAI, AutoGen)

Стейт-машина построена на уровне фреймворка. Модель вызывается как "узел".

- **+**: state management из коробки, визуализация графа.
- **−**: много магии, fine-tune труднее дружить с их форматом, для слабой модели фреймворк будет навязывать паттерны, которым модель не обучена.

### 3.7. Гибрид: MCP для state + SQLite для персистентности

MCP-сервер поверх SQLite. Task_id — первичный ключ, шаги — записи в таблице.

- **+**: рабочий вариант для продакшена, запросы по task_id тривиальны, восстановление после рестарта бесплатно.
- **−**: плюс одна зависимость, схема БД добавляет трение.

## 4. Что реально реализуемо в рамках Дня 6

Напомню ограничения Дня 6:
- 50+ примеров в JSONL (train 80 / eval 20)
- Скрипт валидации
- 10 baseline-ответов без fine-tune
- Критерии оценки
- Клиент загрузки/запуска job в OpenAI API (написать, **не запускать**)

Что из этого **напрямую связано с архитектурой памяти**:

| Артефакт Дня 6 | Зависимость от архитектуры |
|---|---|
| JSONL датасет | **Высокая** — структура `messages` и формат `assistant` определяются выбранной архитектурой. Если выбрали MCP — dataset показывает tool calls с `plan_create`/`step_read`. Если conversation-state — dataset показывает блоки `<STATE>...</STATE>` внутри `assistant`. |
| Скрипт валидации | Средняя — должен проверять наличие ожидаемого паттерна (tool calls, блоков состояния, полей `step_id`, `task_id`). |
| Baseline на eval | Низкая — просто прогон без fine-tune. |
| Fine-tune client (upload + create job) | Низкая — стандартный OpenAI API вызов. |
| Сам MCP-сервер / FS tools | **НЕ обязательно на День 6** — можно написать только прототип tool signatures (мок-реализацию), достаточно для генерации датасета. Полную реализацию можно отложить до Дня 7+. |

**Вывод**: на Дне 6 достаточно **зафиксировать архитектуру и tool signatures**, спроектировать формат датасета под выбранный вариант и сгенерировать 50 примеров. Реальный MCP/FS можно написать как заготовку-стаб.

## 5. Ключевые архитектурные принципы (верны для всех вариантов)

Это те "инварианты", которые обязаны сохраниться в любом из вариантов — иначе идея пользователя не работает:

1. **Task ID как якорь** — любое обращение к состоянию скопировано через `task_id`. Это даёт изоляцию параллельных задач и идемпотентность.
2. **Планирование отделено от исполнения** — в датасете это два разных типа `assistant` ответов (planner-tone vs executor-tone). Модель учится их не смешивать.
3. **Шаг = атомарный микро-контракт** — один шаг содержит: цель, предусловия, действие, самопроверку, критерий завершения, ссылку на следующий шаг.
4. **Перечитывание состояния перед действием** — в датасете каждый executor-ответ **начинается** с вызова `step_read` (или эквивалента). Это самое важное правило для слабой модели.
5. **Self-check материализован в данных** — чек-лист не в промпте, а в файле шага. Модель не проверяет "из головы", она проверяет по выписанному списку.
6. **Фиксированные исходы шага** — `done` / `failed` / `needs_replan`. Нет "частично сделал".
7. **Обязательная ветка `needs_replan`** — ~20% примеров датасета должны показывать, как модель ловит свою ошибку и обновляет план. Без этого получится тупой исполнитель, не способный восстановиться.

## 6. Риски и ограничения

| Риск | Проявление | Митигация |
|---|---|---|
| **Overfit на синтетику** | 40+ примеров сгенерированы одной моделью → модель выучит её стиль, а не задачу | 20% реальных, генерировать синтетику разными моделями (Claude + GPT-4o), разнообразить имена модулей/путей/задач |
| **Раздувание THOUGHT/PLAN** | Модель пишет 500 токенов размышлений на 1 JSON action | В датасете жёсткие лимиты (THOUGHT ≤ 3 предложения), в валидаторе отклонять "болтливые" примеры |
| **Фейковые self-check галочки** | Модель механически ставит `[X]` без реальной проверки | В датасете включить **негативные примеры**, где self-check ловит ошибку и план меняется |
| **"Потеря" task_id** | Модель забыла передать task_id в tool call | Валидатор проверяет наличие task_id во **всех** tool calls, fine-tune JSONL блокирует примеры без task_id |
| **State drift между MCP и реальностью** | Шаг сказал "файл Foo.kt есть", но его удалили в IDE | Вне скоупа Дня 6; решается на уровне MCP (hash-проверка) |
| **Unsupported tool name hallucination** | Модель придумывает несуществующий tool | System prompt жёстко фиксирует разрешённый набор tools, fine-tune закрепляет |

## 7. Критерии оценки (что будем сравнивать baseline ↔ fine-tuned)

### Автоматические метрики (машина проверяет)

1. **Structural compliance** — доля ответов, где есть все обязательные блоки (`task_id`, вызов `step_read` перед action, self-check блок, корректный JSON). Цель: baseline ~30-50%, fine-tuned >90%.
2. **Tool name validity** — доля вызовов несуществующих инструментов. Цель: baseline >5%, fine-tuned ~0%.
3. **Task_id consistency** — во всех сообщениях одной цепочки используется один task_id. Цель: baseline ~60%, fine-tuned 100%.
4. **Self-check presence rate** — есть ли блок самопроверки перед mark_done. Цель: baseline <50%, fine-tuned >95%.
5. **Replan trigger rate** — если tool вернул error, следующий ответ это обновление плана, а не повтор. Цель: baseline ~20%, fine-tuned >80%.

### LLM-as-judge метрики (GPT-4o оценивает по рубрикатору)

6. **Plan quality** — план разбит на атомарные шаги, в каждом шаге есть предусловие и критерий завершения. Шкала 1-5.
7. **Recoverability** — насколько легко человек может вмешаться в середине и продолжить с любого шага. Шкала 1-5.

### Human review

8. **Spot-check на 5 сложных задачах** (multi-file refactoring) — качественное сравнение "разговорности" vs дисциплины.

## 8. Итог анализа

Пользователь предлагает **правильную архитектуру для слабых моделей**: вынести "мышление" из модели в структурированные внешние артефакты и fine-tune-ить модель именно на дисциплину работы с этими артефактами, а не на "быть умнее".

**Что делать на Дне 6**: выбрать одну из 3 архитектур (см. `adv_day6_variants.md`), спроектировать формат датасета под неё, сгенерировать 50 примеров (40 синтетики + 10 реальных), написать валидатор, снять baseline, подготовить upload-client.

**Что не делать на Дне 6**: реализовывать полный MCP-сервер или агент-раннер — это Дни 7-10. Достаточно tool signatures и мока.
