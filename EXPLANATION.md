# EXPLANATION — что это за проект и как он устроен

Файл для того, кто впервые открывает папку `day6/` и хочет разобраться, не читая всего плана на 450 строк. README рядом — про то, *как запустить*. Этот файл — про то, *что здесь происходит и зачем*.

---

## 1. Одним абзацем: что мы делаем

Мы готовим **датасет для fine-tune** маленькой, слабой языковой модели (`gpt-4o-mini`), чтобы она вела себя как **дисциплинированный агент-исполнитель для Kotlin Multiplatform проекта**. Базовая `gpt-4o-mini` знает Kotlin, но когда ей дают задачу «добавь зависимость в `build.gradle.kts`», она либо сразу кидается генерить код, либо путается в tool-вызовах. Наша цель — на fine-tune научить её **сначала составлять план, потом читать шаг из «внешней памяти», потом действовать, потом фиксировать результат**. Каждый раз один и тот же workflow.

Датасет — это 58 примеров диалогов «человек → модель → инструменты → модель → …». Такое обучение называется *supervised fine-tuning*: модель видит идеальный диалог и запоминает паттерн поведения.

---

## 2. Почему это нужно — одна картинка

**Без FT (baseline, реально замерено):**
- Модель в 100% случаев правильно выбирает инструмент `plan_write` первым (это единственное, что она делает хорошо).
- В **0% случаев** пишет блок `THOUGHT:` или `SELF-CHECK:` в контенте сообщения — всё запихивает сразу в аргументы tool_call, что ломает нашу трассируемость.
- На неопределённом запросе («поправь экран настроек — что-то с переключателем не так») **бежит строить план наугад**, вместо того чтобы задать уточняющий вопрос.

**После FT (цель):**
- Те же 100% по plan_write.
- ≥90% примеров с THOUGHT + SELF-CHECK.
- На ambiguous-запросе отвечает блоком `QUESTION: 1) ... 2) ... 3) ...` и ждёт ответа человека.

То есть **FT не учит новым знаниям — он прошивает дисциплину формата.** Это самая типичная и самая рентабельная задача для fine-tune: базовая модель умеет всё нужное, но не умеет это делать *в таком виде*, как нам надо.

## 3. Ключевые понятия (словарик)

- **JSONL** — формат «один JSON на строку», стандарт для обучения ChatGPT-моделей. Каждая строка = один обучающий пример.
- **messages** — массив ходов диалога: `system` (инструкция модели), `user` (вопрос разработчика), `assistant` (ответ модели, возможно с tool_calls), `tool` (результат, который вернул инструмент).
- **tool_call** — когда модель решает не просто ответить текстом, а позвать функцию (например, «прочитай файл `build.gradle.kts`»). Передаётся JSON'ом с `name` и `arguments`.
- **split tool-контракт** — наша архитектурная хитрость: все инструменты модели делятся на **state tools** (управление её собственным планом задачи) и **project tools** (работа с файлами в коде). В B-реализации state tools пишут на диск в `.tasks/<task_id>/`; при переходе на MCP-сервер (вариант A) меняется только их бэкенд — модель остаётся той же.
- **task_id** — уникальный идентификатор задачи, с которым модель оперирует. Приходит через *system prompt*, не через user-сообщение. Модель везде в state-tool вызовах его прописывает.
- **baseline** — замер поведения модели *до* fine-tune. Нужен, чтобы можно было сказать «стало лучше на X%».

## 4. Типы примеров в датасете — три режима

Основной инсайт дня: обучать только на агентских задачах опасно — модель начнёт звать `plan_write` на что угодно, включая простой вопрос «что такое `expect/actual`?». Чтобы этого не случилось, в датасете сознательно три режима:

| Режим | ~% | Зачем | Пример user-реплики |
|---|---|---|---|
| **agent** | 70 | Основной режим: план → шаги → действия | «Добавь Ktor в shared модуль.» |
| **agent_question** | 16 | Учим отказываться и задавать уточнения на неопределённые запросы | «Поправь экран настроек — переключатель какой-то кривой.» |
| **plain** | 14 | Свободная проза на концептуальные вопросы, анти-catastrophic-forgetting | «Чем `expect/actual` отличается от `interface` в KMP?» |

Каждый режим имеет **свой system prompt**. Модель учится распознавать, в каком режиме она сейчас работает, по инструкции в `system`.

## 5. Карта файлов — что за что отвечает

### Корень
| Файл | Что это |
|---|---|
| `README.md` | Инструкция «как запустить» с командной строкой |
| `EXPLANATION.md` | Этот файл — концептуальное объяснение |
| `REPORT.md` | Отчёт по заданию Дня 6 |
| `pyproject.toml` / `requirements.txt` | Python-зависимости (`openai`, `python-dotenv`, `jsonschema`) |
| `.env.example` | Шаблон для API-ключей |

### `contracts/` — что модель может вызывать
- **`tool_schemas.json`** — 9 инструментов в OpenAI tool-format: 5 state tools (`plan_write`, `step_read`, `step_update_result`, `task_status`, `plan_revise`) + 4 project tools (`read_file`, `list_dir`, `search_and_replace`, `write_file`).
- **`step_schema.json` / `plan_schema.json`** — JSON Schema для артефактов, которые state tools пишут на диск.

### `prompts/` — что модели говорят
- **`system_agent.md`** — инструкция агентского режима: список 9 инструментов + правила workflow + слот `<<TASK_ID>>`, который подставляется при генерации.
- **`system_plain.md`** — инструкция plain-режима: «ты senior KMP инженер, отвечаешь прозой, без tool calls».
- **`meta_agent.md` / `meta_question.md` / `meta_plain.md`** — мета-промпты: шаблоны, которые скармливаются генератору, чтобы он произвёл новый пример нужного режима.

### `dataset/` — сами данные
- **`seeds/`** — 12 примеров, написанных вручную. Из них два (`golden_01/02`) — эталонные образцы с replan-веткой и happy-path. Остальные покрывают каждый тип задачи (develop × 3 / refactor / bugfix × 2 / research × 2 / tests + question × 2 + plain).
- **`synthetic/`** — 46 примеров, сгенерированных через LLM по мета-промптам.
- **`scenarios.py`** — «банк идей»: списки сценариев разного типа (ADD_DEPENDENCY, BUGFIX, RESEARCH, …) плюс пулы имён модулей/библиотек для вариативности.
- **`gen_synthetic.py`** — генератор: берёт сценарий из банка, подставляет в мета-промпт, отправляет в LLM, валидирует результат, сохраняет в `synthetic/`. Retry-with-feedback: если валидация не прошла, в следующую попытку добавляются конкретные ошибки.
- **`build_seeds.py`** — утилита превращения `seeds/*.json` в JSONL для скармливания в тренировку.
- **`mix_and_split.py`** — микс `seeds + synthetic` → **train.jsonl (47)** + **eval.jsonl (11)**, stratified 80/20 (пропорции режимов сохраняются в обеих частях).
- **`summarize.py`** — финальная справка: что получилось после mix_and_split.
- **`train.jsonl` / `eval.jsonl`** — итог, то что пойдёт в OpenAI FT API.
- **`eval_plain.jsonl`** — 5 концептуальных вопросов для отдельной проверки, не сломался ли модель на plain-режиме после FT (anti-catastrophic-forgetting).

### `validator/` — проверка корректности
- **`validate.py`** — три прохода: **structural** (валидный JSON, роли на месте, непустые content), **semantic** (task_id во всех state-tool вызовах, `plan_write` первым, read-before-action, THOUGHT+SELF-CHECK), **dedup** (одинаковые user-тексты). Запускается как `python -m validator.validate <path>`.

### `baseline/` — замер без FT
- **`run_baseline.py`** — прогоняет (system + первый user) каждого seed через `gpt-4o-mini` БЕЗ fine-tune и собирает авто-метрики.
- **`outputs/summary.md`** — зафиксированные результаты baseline. Это наша «точка отсчёта» для будущего сравнения.

### `criteria/` — что значит «стало лучше»
- **`criteria.md`** — таблица из 5 авто-метрик и 2 LLM-judge метрик с численными целями. Каждая метрика измерима тем же `validate.py` или отдельным judge-скриптом.

### `ft_client/` — клиент для OpenAI fine-tune
- **`upload.py`** — `client.files.create(file, purpose="fine-tune")`.
- **`create_job.py`** — создание FT-задачи. По умолчанию **dry-run** (печатает что бы отправил, ничего не отправляет). Реальная отправка требует флага `--confirm`.
- **`poll.py`** — цикл `client.fine_tuning.jobs.retrieve(job_id)` с 30-секундным интервалом.
- **`README.md`** — команды и порядок использования.

## 6. Пайплайн по шагам — для новичка

### Шаг 1. Контракты и prompts (что модель видит)
Решаем, какие инструменты у модели есть и как её представлять ей задачу. Всё это **пишется один раз**: 9 инструментов, 2 system prompt'а, 3 мета-промпта.

### Шаг 2. Seeds (12 идеальных примеров)
Руками пишутся 12 полных диалогов: плохо написанный seed = 46 плохих клонов. Каждый содержит все паттерны, которые мы хотим видеть в модели: `plan_write` первым, `step_read` перед action, THOUGHT + SELF-CHECK, один из seeds — с replan-веткой (демонстрация восстановления после ошибки). 12 seeds / 58 total = 20.7% «реальных» — удовлетворяет минимальной доле по заданию.

### Шаг 3. Генератор синтетики (46 примеров)
Для каждого нового примера: (а) выбираем случайный сценарий из `scenarios.py`, (б) подбираем seed того же типа как reference, (в) кидаем всё в мета-промпт, (г) отправляем в LLM, (д) парсим ответ, (е) валидируем через `validate.py`, (ж) если ошибка — ретраем с обратной связью, (з) кладём в `synthetic/`. Вариативность достигается случайным выбором имени модуля / библиотеки / стиля отступов.

### Шаг 4. Валидатор
Каждый файл проверяется на 30+ условий. Фильтр на дубли user-текста (если синтетика выдала близкий или одинаковый запрос). Те, что прошли — идут в микс, те, что не прошли — в отчёт об ошибках.

### Шаг 5. Mix + Split
Seeds и synthetic объединяются. Случайный seed=42 для воспроизводимости. Stratified split: пропорции режимов (70/16/14) сохраняются в train и eval. Итог: `train.jsonl` (47 строк) + `eval.jsonl` (11 строк).

### Шаг 6. Baseline (замер ДО FT)
Берём те же seeds и прогоняем через `gpt-4o-mini` без fine-tune. Смотрим: какой инструмент она зовёт первым? Есть ли THOUGHT в content? Есть ли task_id в state-tool args? Результат → `baseline/outputs/summary.md`.

### Шаг 7. Критерии
Пишем, какие числа на eval мы хотим увидеть после FT. Для каждой метрики — baseline-значение и target-значение. Это наш contract с самим собой: «fine-tune считается успешным, если X ≥ Y».

### Шаг 8. FT-клиент
Три скрипта: загрузить `train.jsonl` в OpenAI, создать fine-tune job, опрашивать статус. `create_job.py` сознательно **не запускается без `--confirm`**, чтобы случайно не потратить деньги.

## 7. А что с результатом?

В папке получается:
- **Готовые данные для обучения** (`train.jsonl`, `eval.jsonl`, `eval_plain.jsonl`)
- **Воспроизводимый инструмент генерации** (`gen_synthetic.py`) — можно нарастить датасет за минуты.
- **Валидатор** — можно запускать на любом JSONL, своём или чужом.
- **Численная метрика baseline** — точка отсчёта.
- **Критерии «стало лучше»** — что именно считаем успехом.
- **Клиент fine-tune** — готов к запуску одной командой с `--confirm`.

Следующий день (День 7+) — запуск реального FT, замер метрик, сравнение с baseline. Если всё ок — подключение к реальному KMP-проекту через MCP Memory Layer (вариант A из варианта-плана).

---

## 8. Почему B, а не A

См. `plans/adv_day6_variants.md`. Коротко: вариант A (MCP Memory Layer) требует рабочего MCP-сервера *до* того, как мы можем сгенерировать хоть одну строку датасета. Вариант B (file-scratchpad) даёт работающий датасет за день, а архитектура инструментов такая, что **при переходе B → A модель не переобучается** — меняется только реализация state-tools на бэкенде.

---

## 9. Если что-то непонятно

- Почему 12 seeds? Референс для генератора + обязательная минимальная доля «реальных» (≥20% от 58 = 12). Каждый seed — полный образец тренируемого паттерна, на котором размножаются 46 синтетических.
- Можно ли делать без split? Нет — eval нужен для оценки FT, иначе мы не сможем сказать, стало лучше или хуже.
- Зачем plain-примеры? Чтобы модель не разучилась отвечать на концептуальные вопросы после FT на агентских задачах.
- Почему `gpt-4o-mini` как target? Маленькая, дешёвая, OpenAI её официально разрешает fine-tune'ить. Для локального FT на Mac см. комментарий в `REPORT.md` — там мы рассматриваем замену на open-weight модель (Qwen2.5-7B, Llama-3.1-8B) без изменения датасета.
