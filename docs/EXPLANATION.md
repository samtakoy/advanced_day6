# EXPLANATION — что это за проект и как он устроен

Файл для того, кто впервые открывает папку `day6/` и хочет разобраться. README рядом — про то, *как запустить*. Этот файл — про то, *что здесь происходит и зачем*.

---

## 1. Одним абзацем: что мы делаем

Мы готовим **датасет для fine-tune** языковой модели, чтобы она умела **извлекать структурированное описание задачи из свободного текста**. На входе — Slack-тикет или обсуждение в чате ("надо разгрести StocksDatabase, он инжектится напрямую во все репозитории..."). На выходе — JSON с полями: название, тип, блок роадмапа, затронутые модули, зависимости, критерии приёмки, out of scope.

Это задача **extraction**, не classification и не generation. Модель не придумывает — она извлекает структуру из текста, маппя на таксономию конкретного проекта (KMP stocks).

---

## 2. Почему это нужно — одна картинка

**Без FT (baseline, реально замерено):**
- Модель путает модули — пишет `core-features/stocks` вместо алиаса `cf-stocks`, выдумывает `cf-theme` (нет такого)
- modules IoU ≈ 0.46 — почти половина модулей неправильные
- Часто оборачивает JSON в markdown или добавляет текст до/после
- 2 из 11 eval-примеров не парсятся как JSON

**После FT (реально замерено):**
- JSON parse: 100% (каждый ответ — валидный JSON)
- 0 ошибок валидации схемы
- Block match: 33% → 73%
- AC recall: 0.03 → 0.45

**FT не учит новым знаниям — он прошивает таксономию проекта.** Базовая модель умеет извлекать структуру, но не знает наших алиасов модулей и блоков.

---

## 3. Ключевые понятия

- **JSONL** — формат «один JSON на строку», стандарт для обучения. Каждая строка = один обучающий пример.
- **messages** — массив из 3 элементов: `system` (инструкция + таксономия), `user` (свободное описание задачи), `assistant` (эталонный JSON-ответ).
- **modules** — алиасы модулей проекта из таблицы в system prompt (16 штук: `m-main`, `cf-stocks`, `db`, `uikit` и т.д.).
- **newModules** — полные пути для модулей вне таблицы (создаются задачей: `composeApp`, `modules:core:telemetry`).
- **gold.md** — source of truth: 56 эталонных JSON-ответов с маркерами [TRAIN]/[EVAL]. Все правки — сначала в markdown, потом пересборка JSONL.
- **baseline** — замер поведения модели *до* fine-tune. Нужен для сравнения.

---

## 4. Что внутри датасета

56 примеров (45 train / 11 eval), стратифицированных по 6 блокам:

| Блок | Описание | Примеров |
|------|----------|----------|
| `workspace_foundation` | БД, навигация, CRUD воркспейсов | 9 |
| `indicators` | Расчёт индикаторов, конфиги | 6 |
| `analysis` | ExperimentRunner, бэктест, метрики | 8 |
| `polish_and_glue` | Кросс-навигация, экспорт/импорт, uikit | 10 |
| `breadth` | Yahoo/Binance, Watchlist, Alerts, Portfolio | 13 |
| `tech_debt_refactor` | Рефакторинги, изоляция библиотек | 10 |

Каждый пример — single-turn: system prompt с таксономией → свободное описание задачи → эталонный JSON.

## 5. Как устроен source of truth

Содержание живёт в **markdown**, не в JSONL:

```
data/extraction/
  system.md              — system prompt (единый для всех)
  gold.md                — 56 gold-JSON с маркерами [TRAIN]/[EVAL]
  tasks1_prose.md        — user-входы для задач 1-25
  tasks2.md              — user-входы для задач 26-50
  tasks_adversarial.md   — adversarial user-входы 51-56
```

Причина: 56 gold-JSON легче ревьюить в markdown-блоках, чем в JSONL. При правке — правим markdown, пересобираем JSONL одной командой:

```bash
python -m src.dataset.build_dataset
```

## 6. Extraction-схема

```json
{
  "title": "string",
  "type": "feat | refactor | research",
  "block": "workspace_foundation | indicators | analysis | polish_and_glue | breadth | tech_debt_refactor",
  "modules": ["алиас из таблицы"],
  "newModules": ["полный путь для новых модулей"],
  "dependsOn": [1, 2],
  "acceptanceCriteria": ["проверяемое условие"],
  "outOfScope": ["явно исключённое"]
}
```

Правила:
- `modules` — только явно упомянутые в тексте, только из таблицы алиасов
- `newModules` — полные пути для модулей вне таблицы
- `acceptanceCriteria` — только проверяемые условия (gradle, grep, smoke)
- `outOfScope` — только явно названное как «не делаем»
- Неупомянутое → пустой массив `[]`

---

## 7. Метрики

| Метрика | Что измеряет | Как считается |
|---------|-------------|---------------|
| `type` / `block` | Правильность классификации | Exact match |
| `modules IoU` | Правильность модулей | Jaccard similarity множеств |
| `newModules IoU` | Правильность новых модулей | Jaccard similarity |
| `dependsOn IoU` | Правильность зависимостей | Jaccard similarity |
| `AC recall` | Покрытие критериев | Доля gold-критериев в ответе |
| `OoS precision` | Точность out-of-scope | Доля корректных в ответе |
| `schema valid` | Соответствие схеме | Валидатор (типы, enum, дубли) |

**Главная метрика** — `modules IoU`. Baseline models путают пути/классы с алиасами модулей.
